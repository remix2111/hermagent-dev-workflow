# Hermes 开发工作流

完整的开发方法论：从想法验证、计划制定、委派执行，到系统调试和失败恢复。五个阶段顺序推进，每个阶段有明确的进出标准。

```
Spike ──→ Plan ──→ Execute ──→ Debug ──→ Recover
验证想法    制定计划    委派执行    系统调试    失败恢复
```

---

## 阶段一：Spike（探索实验）

**目的：** 在投入正式开发前，用可丢弃的代码验证想法的可行性。

### 何时使用

- 想快速验证某个技术方案是否可行
- 需要比较多种实现方案（A vs B）
- 存在不确定的未知因素，无法通过文档或阅读代码解决

### 何时不用

- 答案可以从文档或阅读代码中获得 → 直接研究，无需写代码
- 已经是生产级开发 → 进入 Plan 阶段
- 想法已经过充分验证 → 直接进入实现

### 核心方法：探索循环

```
分解 ──→ 研究 ──→ 构建 ──→ 结论
  ↑__________________________↓
         根据发现迭代
```

#### 1. 分解

将用户的想法分解为 **2-5 个独立的可行性问题**，每个问题为一个 Spike，按风险从高到低排列：

| # | Spike | 验证目标 (Given/When/Then) | 风险 |
|---|-------|--------------------------|------|
| 001 | websocket-streaming | Given WS 连接, When LLM 流式输出, Then 客户端 <100ms 收到 chunk | 高 |
| 002a | pdf-parse-pdfjs | Given 多页 PDF, When 用 pdfjs 解析, Then 提取结构化文本 | 中 |
| 002b | pdf-parse-camelot | Given 多页 PDF, When 用 camelot 解析, Then 提取结构化文本 | 中 |

**Spike 类型：**
- **标准型** — 一种方案回答一个问题
- **对比型** — 同一问题不同方案（相同编号，字母后缀 a/b/c）

好的 Spike 问题：具体、可观察输出。差的：太宽泛、无观察输出、或只是"去读文档"。

#### 2. 研究

每个 Spike 在写代码前先做快速研究：
- 列出可选方案（工具/库），比较维护状态、优缺点
- 选择一种方案并说明理由
- 纯逻辑问题（无外部依赖）可跳过

#### 3. 构建

每个 Spike 一个独立目录，保持简单：

```
spikes/
├── 001-websocket-streaming/
│   ├── README.md
│   └── main.py
└── 002a-pdf-parse-pdfjs/
    ├── README.md
    └── parse.js
```

构建原则：
- 优先做可交互的输出（CLI、HTML 页面、Web 端点），而非仅日志
- 深度优先 — 测试边界条件，追踪意外发现
- 避免复杂配置，硬编码一切
- 对比 Spike（a/b/c）可用 `delegate_task` 并行执行

#### 4. 结论

每个 Spike 的 README.md 以结论结尾：

```markdown
## 结论：VALIDATED | PARTIAL | INVALIDATED

### 可行的
- ...

### 不可行的
- ...

### 意外发现
- ...

### 对正式开发的建议
- ...
```

- **VALIDATED** — 核心问题已回答"是"，有证据支持
- **PARTIAL** — 在特定约束下可行，记录约束条件
- **INVALIDATED** — 不可行，这是成功的结果（避免了走弯路）

对比型 Spike 需做并排对比并选出优胜方案。

## 阶段二：Plan（制定计划）

**目的：** 编写可执行的实现计划，让实现过程不需要猜测。

### 核心原则

> **好的计划让实现显而易见。如果需要猜测，计划就不完整。**

### Plan 模式行为

- 只做规划，不执行代码
- 不修改项目文件（计划文件除外）
- 可做只读探索来收集上下文
- 产出物：保存到 `.hermes/plans/YYYY-MM-DD_HHMMSS-<slug>.md`

### 任务粒度

**每个任务 = 2-5 分钟的专注工作。**

每步一个动作：
- "写失败测试" → 一步
- "运行确认失败" → 一步
- "写最小实现" → 一步
- "运行确认通过" → 一步
- "提交" → 一步

### 计划文档结构

```markdown
# [功能名称] 实现计划

> **给 Hermes：** 使用 subagent-driven-development 按任务逐个实现。

**目标：** [一句话描述]

**架构：** [2-3 句方案说明]

**技术栈：** [关键技术/库]

---

### 任务 1: [描述性名称]

**目标：** 此任务完成什么（一句话）

**文件：**
- 新建: `exact/path/to/new_file.py`
- 修改: `exact/path/to/existing.py:45-67`
- 测试: `tests/path/to/test_file.py`

**步骤 1：写失败测试**

\```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
\```

**步骤 2：运行确认失败**

运行: `pytest tests/test.py::test_specific_behavior -v`
预期: FAIL

**步骤 3：写最小实现**

\```python
def function(input):
    return expected
\```

**步骤 4：运行确认通过**

运行: `pytest tests/test.py::test_specific_behavior -v`
预期: PASS

**步骤 5：提交**

\```bash
git add tests/test.py src/file.py
git commit -m "feat: add specific feature"
\```
```

### 编写流程

1. **理解需求** — 阅读功能需求、设计文档、验收标准、约束条件
2. **探索代码库** — 用 `search_files`/`read_file` 了解项目结构、相似功能和现有测试
3. **设计方案** — 决定架构模式、文件组织、依赖、测试策略
4. **编写任务** — 按顺序：基础设施 → 核心功能（TDD） → 边界条件 → 集成 → 清理
5. **补充细节** — 精确路径、完整代码、确切命令、验证步骤
6. **审查计划** — 检查任务顺序、粒度、路径、代码完整性

### 编写原则

| 原则 | 含义 |
|------|------|
| **DRY** | 不重复自己 — 提取公共逻辑 |
| **YAGNI** | 你不会需要它 — 只实现当前需要的 |
| **TDD** | 测试驱动 — 每个任务包含完整的红→绿→重构周期 |
| **频繁提交** | 每个任务完成后提交 |

### 常见错误

- **模糊任务：** "添加认证" → 应为 "创建 User 模型，包含 email 和 password_hash 字段"
- **不完整代码：** "添加验证函数" → 应附带完整函数代码
- **缺少验证：** "测试一下" → 应为 "运行 `pytest tests/test_auth.py -v`，预期 3 passed"
- **缺少路径：** "创建模型文件" → 应为 "创建 `src/models/user.py`"

## 阶段三：Execute（执行委派）

**目的：** 将计划转化为实际代码，通过子代理委派执行。

### 执行交接

计划完成后，使用 `subagent-driven-development` 技能执行：
- 每个任务分配一个独立的 `delegate_task`
- 每个任务完成后先做规格符合性审查
- 规格通过后做代码质量审查
- 两项审查均通过才进入下一个任务

```
Plan 完成 → delegate_task(任务1) → 审查 → delegate_task(任务2) → 审查 → ...
```

## 阶段四：Debug（系统调试）

**目的：** 用系统化的方法找到根因而非修复症状。

### 铁律

```
没有根因调查，不得进行任何修复。
```

### 四阶段调试法

#### Phase 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息** — 堆栈跟踪、行号、错误码，不跳过任何警告
2. **构建紧反馈循环** — 一个命令即可复现、快速、确定性的测试：
   - 优先：失败单元测试 → HTTP/curl 脚本 → CLI 调用 → 浏览器脚本
   - 对于不稳定 bug：提高复现率（重复 100 次、加压力、缩小时间窗口）
3. **检查最近变更** — `git log --oneline -10`、`git diff`
4. **多组件系统收集证据** — 在每个组件边界记录数据进出
5. **追踪数据流** — 坏值从哪里来？向上游追溯直到找到源头

**完成标准：** 理解了"是什么"和"为什么"，可以陈述并测试根因假设。

#### Phase 2：模式分析

1. **最小化复现** — 一次删减一个元素，只保留触发失败所必需的
2. **找可工作的例子** — 在同一代码库中找类似但正常工作的代码
3. **对比差异** — 列出工作与不工作之间的每一个差异
4. **理解依赖** — 需要什么组件、配置、环境、假设

#### Phase 3：假设与测试

1. **形成可证伪假设** — 生成 3-5 个假设，按可能性和证伪成本排序
2. **最小化测试** — 一次只改变一个变量，优先用断点而非大量日志
3. **验证后继续** — 有效 → Phase 4；无效 → 形成新假设，不要叠加修复

#### Phase 4：实施修复

1. **先写回归测试** — 最简单复现，必须在修复前写好
2. **单一修复** — 只改一处，不做"顺便"改进
3. **验证修复** — 运行回归测试 + 完整测试套件
4. **三次法则** — 如果修复失败：
   - < 3 次：回到 Phase 1，带着新信息重新分析
   - **≥ 3 次：停止，质疑架构** — 每次修复都暴露新的耦合/状态问题 → 可能是架构问题

### 红旗信号

以下思维出现时立即停止，回到 Phase 1：
- "快速修复一下，之后再调查"
- "试试改 X 看看行不行"
- "跳过测试，手动验证"
- "可能是 X，让我修一下"
- "我不完全理解但这可能有效"
- "再试一次"（已经试过 2+ 次）
- 每次修复都暴露新问题

### 常见借口 vs 现实

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因，流程对简单 bug 也很快 |
| "紧急情况，没时间走流程" | 系统调试比猜测+试错更快 |
| "先试试这个，再调查" | 第一次修复定调，一开始就做对 |
| "一次改多处省时间" | 无法隔离有效因素，还会引入新 bug |

## 阶段五：Recover（失败恢复）

**目的：** 当 `delegate_task` 返回失败时，自动分析并决定重试策略。

### 触发条件

当 `delegate_task` 调用返回失败状态时自动触发。

### 分析步骤

1. **读取 exit_reason** — 理解子代理为何退出（超时、错误、资源不足等）
2. **审查 tool_trace** — 检查子代理的工具调用记录，定位失败点
3. **分类失败类型：**
   - **可重试错误** — 网络超时、临时资源不可用、竞态条件
   - **需调整策略** — 工具选择不当、上下文不足、任务描述模糊
   - **不可重试** — 权限不足、数据损坏、不支持的平台/环境

### 重试策略

| 情况 | 策略 |
|------|------|
| 可重试错误 | 原样重试，最多 1 次 |
| 需调整策略 | 修改 task 描述（补充上下文、更换工具集、缩小范围）后重试，最多 1 次 |
| 不可重试 | 不再重试，向用户报告失败原因和建议 |

### 规则

- 最多重试 **1 次**
- 重试时必须在日志中说明：失败原因 → 调整了什么 → 重新委派
- 两次都失败则停止，向用户清晰报告

## 快速参考

| 阶段 | 目的 | 关键动作 | 产出 |
|------|------|----------|------|
| Spike | 验证可行性 | 分解→研究→构建→结论 | `spikes/NNN-name/` 含 README + 结论 |
| Plan | 制定计划 | 理解→探索→设计→编写任务 | `.hermes/plans/<timestamp>.md` |
| Execute | 委派执行 | delegate_task 逐任务执行+审查 | 可工作的代码 |
| Debug | 系统调试 | 4 阶段根因分析→单一修复 | 回归测试 + 根因修复 |
| Recover | 失败恢复 | 分析 exit_reason → 决定重试/放弃 | 重试后成功或清晰失败报告 |
